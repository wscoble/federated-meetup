# Security audit — 2026-07-06 — wscoble/federated-meetup @ 8d3c07d

## Executive summary

The 2026-07-05 anti-dox fix on `/my-rsvps` and `/my-rsvps/cancel` holds
under the tests in `internal/web/anti_dox_test.go`. The nine regression
tests cover IDOR-by-email, IDOR-by-(email+group+event), magic-link
single-use, invalid-token non-enumeration, per-IP rate limit, expired
session, and reissue-overwrite. They pin a sound model: the rsvp_token /
session_token is the credential; the email is just routing.

Outside that flow, the surface has multiple unauthenticated or
broken-authentication paths that the 2026-06-30 audit (C-1..C-6, H-1..H-10)
did not cover because the host/ActivityPub/ConnectRPC/email layers were
largely stubbed then. The largest gaps today are: (1) HTTP signature
verification on the ActivityPub inbox is a no-op — every signed or
unsigned POST is accepted; (2) ConnectRPC has zero authentication, so
SubmitTransition / SubmitUserAction are open to any caller that can
reach the daemon; (3) dashboard endpoints gate on the authenticated
`group_key` but never verify that the supplied `event_id` actually
belongs to that group, so an organizer of group A can read CSV, cancel,
and check-in on group B's events; (4) the freshly-generated organizer
token is logged in plaintext at creation; (5) both session and CSRF
cookies are `Secure: false`; (6) `/metrics` is exposed unauthenticated.

Findings: **6 CRITICAL, 9 HIGH, 12 MEDIUM, 9 LOW**.

## CRITICAL

### C-1. ActivityPub inbox accepts unsigned AND unverified-signed POSTs
- **Location:** `internal/activitypub/signatures.go:56-68`,
  `internal/activitypub/activitypub.go:255-307,318-323`.
- **Attack:** `handleInbox` calls `VerifyHTTPSignature` only when a
  `Signature` header is present. The function explicitly does
  `return nil` for missing signatures (the v0 "permissive mode" note in
  the SECURITY.md "Application Security" section). When a signature IS
  present, `VerifyHTTPSignature` parses the header but never fetches
  the actor's public key, never checks the digest, never verifies the
  Ed25519/RSA signature. The TODO at line 65 confirms this. `handleInbox`
  also trusts the `actor` field in the JSON body to add a follower with
  no proof the actor controls the claimed keyId.
- **Impact:** Anyone on the internet can POST a `Follow` to
  `/ap/inbox/{group_name}` and be added to the followers list. Because
  `DeliverNewEvent` iterates `a.followers[name]` and POSTs the full
  event to each follower's claimed inbox URL (`activitypub.go:423-447`),
  an attacker can become a follower of any group and then SSRF the
  daemon: when the next event is created, the daemon POSTs the event
  payload to attacker-chosen URLs (`DiscoverInboxURL` follows the
  attacker-supplied actor URL with no allowlist). This is also a
  stored-data attack: bogus followers persist in the in-memory map and
  poison every subsequent delivery.
- **Recommendation:** Implement the TODO. For unsigned requests, reject
  with 401. For signed requests, fetch the actor document at
  `keyId`, verify the public key matches the document, recompute the
  digest, and verify the signature. Add a per-inbox rate limit and a
  max body size (e.g. 1 MiB) before `io.ReadAll`.
- **Effort:** 1–2 days. Reuse `internal/crypto` (Ed25519) for verification.

### C-2. ConnectRPC SubmitTransition / SubmitUserAction have no authentication
- **Location:** `internal/host/service.go:251-282,289-360,409-464,471-504,
  514-566,587-657`.
- **Attack:** All seven RPC handlers (GetGroup, GetEvent, ListEvents,
  ListGroups, SubmitTransition, SubmitUserAction, ResolveName, plus the
  mirror-side GetLog, GetSnapshot, SubmitEvidence, Subscribe) are exposed
  to any HTTP caller. There is no interceptor, no per-method
  authorization, no auth header check. `SubmitUserAction` additionally
  logs `WARN: ... user envelope signature verification skipped (v0)`
  and does not verify `r.UserEnvelope` at all (`service.go:315-318`).
- **Impact:** An unauthenticated remote attacker who can reach the
  ConnectRPC port can: (a) read every group's full state via
  GetGroup/GetSnapshot; (b) submit arbitrary transitions if they can
  produce valid steward signatures (this is enforced at the protocol
  layer, so impact is "leak all state" rather than "write all state");
  (c) submit arbitrary user actions (RSVP, CANCEL_RSVP, ATTEST) on
  behalf of any user, because the user envelope is not checked.
  Activity (c) is fully exploitable today: a single POST to
  `SubmitUserAction` with `type=RSVP`, `transition_payload=<proto>,`
  and any `user_envelope` (or none) will apply the RSVP to the chosen
  group's state.
- **Recommendation:** Add a `connect.Interceptor` that requires a bearer
  token / mTLS for the write RPCs (SubmitTransition, SubmitUserAction,
  SubmitEvidence). Verify the user envelope against a registered user
  key (re-open M-3 from the prior audit: `EquivocationEvidence
  TransitionA/B` populating is a separate but related gap). At minimum,
  close the `SubmitUserAction` user-envelope gap before any production
  exposure.
- **Effort:** 2–3 days for the interceptor; 1 day for the envelope
  verification.

### C-3. Dashboard endpoints do not verify event_id belongs to group_key
- **Location:** `internal/web/handlers.go:1282-1311 (handleCheckIn)`,
  `1444-1474 (handleCancelEvent)`, `1639-1681 (handleAttendeesCSV)`,
  `149-156 (route)`.
- **Attack:** `isAuthenticated(r)` returns the caller's `group_key`
  from the session cookie. The handlers then read `event_id` from the
  URL/form and act on it directly: `s.store.ListRsvpsForEvent(groupKey,
  eventID)` for CSV export; `s.store.MarkRsvpAttended(eventID, email)`
  for check-in; for cancel, `s.product.Store().GetEvent(eventID)` /
  `s.store.GetEvent(groupKey, eventID)`. None of these lookups verify
  that the event_id actually belongs to the authenticated group_key.
  An organizer of group A can: export the full attendee list (names +
  emails + check-in status) of any event in any group; cancel any
  event; check in any attendee in any event.
- **Impact:** Cross-group PII leak + sabotage. An organizer of a
  throwaway group can scrape attendee lists of every event in the
  system. Combined with the dox-by-attendance threat this is the same
  attack surface the 2026-07-05 fix was meant to close, but on the
  organizer side instead of the attendee side.
- **Recommendation:** In each handler, after fetching the event, verify
  `e.GroupID == groupKey` (or for product-store lookups, verify the
  group matches). Reject with 404 on mismatch (don't disclose the
  event exists). Add a regression test per endpoint.
- **Effort:** 0.5 day.

### C-4. Organizer token logged in plaintext at group creation
- **Location:** `internal/web/handlers.go:963-964`.
  ```go
  log.Printf("web: group created: %s (%s) — organizer: %s <%s>",
      groupKey, displayName, organizerName, organizerEmail)
  log.Printf("web: organizer token for %s: %s", groupKey, organizerToken)
  ```
- **Attack:** Anyone with read access to the host's logs (operator,
  log-aggregator, accidentally-shipped journal, k8s log viewer) can
  take over any group. The token is the only credential guarding the
  dashboard.
- **Impact:** Full organizer compromise for every group whose
  creation has been logged. The k8s deploy configures
  `readOnlyRootFilesystem: true` (per SECURITY.md) but logs are still
  shipped to stdout; any log-aggregation sidecar that captures stdout
  captures the token.
- **Recommendation:** Remove the `organizerToken` log line. The token
  must be shown to the user exactly once (e.g. on a "save this token,
  it will not be shown again" page after group creation) and never
  written to logs. If a fingerprint is needed for audit, log the
  truncated hash (Fingerprint is already in `internal/crypto`).
- **Effort:** 0.25 day.

### C-5. Session and CSRF cookies set `Secure: false`
- **Location:** `internal/web/middleware.go:68-77` (CSRF cookie),
  `159-168` (session cookie).
- **Attack:** Both cookies are issued with `Secure: false`. If the
  site is ever served over plain HTTP (operator misconfig, dev URL,
  internal proxy), the session token and CSRF token are transmitted
  in cleartext. A network observer can replay the session cookie to
  hijack the organizer's dashboard and CSRF cookie to forge requests.
  Even with the Cloudflare-Tunnel-everywhere posture, an
  origin-side misconfig bypasses the tunnel's TLS guarantee.
- **Impact:** Session hijack + CSRF bypass if any HTTP traffic ever
  reaches the listener. The k8s service is ClusterIP-only, but the
  LocalReverseProxy fallback or a future ingress change would
  immediately expose the secret.
- **Recommendation:** Set `Secure: true` on both cookies; add
  `Strict-Transport-Security: max-age=63072000; includeSubDomains`
  to the security-headers middleware. Make the `Secure` flag
  configurable from a single env var so dev-mode is still possible.
- **Effort:** 0.25 day.

### C-6. AP inbox follower injection + delivery SSRF
- **Location:** `internal/activitypub/activitypub.go:255-343,
  404-447`.
- **Attack:** Companion to C-1. Even after the signature verification
  is fixed, `handleInbox` adds `actor` to `a.followers[name]` on any
  `Follow` activity. `DeliverActivity` then `httpClient.Do`s to the
  URL the attacker advertised. This is a server-side request forgery
  vector: the daemon can be made to POST to internal IPs (e.g.
  `http://169.254.169.254/...` on AWS, internal k8s services, the
  host's own metadata endpoint). The httpClient has a 30s timeout but
  no IP allowlist and no scheme pin.
- **Impact:** SSRF; can be used to probe internal services, hit
  cloud-metadata endpoints, and exfiltrate data via the response
  body (which is currently discarded but could be reflected in
  follow-redirect behaviour).
- **Recommendation:** After C-1 is fixed, also: (a) store followers
  as keyId URLs and resolve their inbox at delivery time; (b)
  resolve-and-fetch the inbox URL with an HTTP client that refuses
  private/loopback IPs and HTTP redirects; (c) cap the followers
  set per group.
- **Effort:** 1 day.

## HIGH

### H-1. CSRF cookie is not HttpOnly
- **Location:** `internal/web/middleware.go:72`.
- **Attack:** `HttpOnly: false` means any XSS on the site can read the
  CSRF token from `document.cookie`. Combined with `Secure: false`,
  the CSRF token (and the session token, which IS HttpOnly — but only
  the session) can be exfiltrated by a malicious script. The current
  CSP (`script-src 'self'`) blocks inline script but does not protect
  against a future XSS in a vendored script.
- **Impact:** Reduces defense-in-depth; required if a CSP bypass is
  ever found.
- **Recommendation:** Set `HttpOnly: true` on the CSRF cookie. The
  double-submit pattern does not need JS access to the cookie; the
  form field is what is compared.
- **Effort:** 0.1 day.

### H-2. `/metrics` exposed unauthenticated
- **Location:** `internal/web/server.go:157`,
  `internal/web/metrics.go:155-189`.
- **Attack:** `/metrics` is registered on the public mux with no auth.
  Anyone on the internet can read Prometheus metrics: per-route
  request counts, average durations, process start time. The current
  metrics do not leak PII (no emails, no IPs), but they DO leak
  request volume per route — useful for an attacker to time a
  brute-force campaign ("/dashboard/login success rate is 0.001%
  per IP, so I need 1000 IPs"). Future metrics additions may
  accidentally include per-user counters.
- **Impact:** Reconnaissance + future PII-leak risk.
- **Recommendation:** Either bind `/metrics` to a separate listener
  (ClusterIP-only, scrape from inside the cluster), or require
  `Authorization: Bearer <METRICS_TOKEN>` from an env var.
- **Effort:** 0.5 day.

### H-3. `/checkout/{order_id}` has no authorization
- **Location:** `internal/web/handlers.go:1316-1373`.
- **Attack:** The checkout page renders the buyer's email, the event
  title, the amount, and the Stripe session URL with no authentication.
  The `order_id` is `generateToken()[:16]` (handlers.go:393), which
  is only 64 bits of entropy — short enough that a determined
  attacker with the daemon's URL can enumerate (2^64 attempts is
  infeasible, but the per-order timestamp + 64-bit space + a
  background crawler of likely prefixes is a non-trivial risk for
  a small site).
- **Impact:** Email + event + price leak for every order. Brute-force
  feasibility depends on how predictable the order_id is in practice
  (hex of 16 random bytes → 64 bits, full CSPRNG, so direct brute
  force is hard; the more likely vector is leaked order_ids from
  email, logs, referer headers).
- **Recommendation:** Require a signed cookie or a magic-link token
  to view `/checkout/{order_id}`. The Stripe session URL should
  never be rendered in plain HTML for an unauthenticated user.
  Use 256 bits for order_id (`generateHighEntropyToken`).
- **Effort:** 0.5 day.

### H-4. `To:` header CRLF injection in SMTPSender
- **Location:** `internal/email/sender.go:118-129`.
- **Attack:** `buildMessage` does `fmt.Fprintf(&b, "To: %s\r\n", to)`
  with no validation. If the `to` value (which comes from
  `r.FormValue("email")` in `handleRsvpSubmit` etc., validated only
  for length) contains `\r\nBcc: attacker@x.com`, the resulting email
  has injected headers. The current `validateString` checks length
  but not charset; a value of `victim@x.com\r\nBcc: attacker@x.com`
  passes.
- **Impact:** Email header injection → arbitrary additional
  recipients, subject injection, body injection on vulnerable
  clients.
- **Recommendation:** Validate `to` is a single RFC 5322 address
  (or use a parser like `net/mail.ParseAddress`). Strip
  `\r` and `\n` from any value that lands in a header.
- **Effort:** 0.25 day.

### H-5. AP inbox has no body size limit
- **Location:** `internal/activitypub/activitypub.go:286`.
- **Attack:** `io.ReadAll(r.Body)` with no `http.MaxBytesReader`
  wrapper. A single POST with a multi-GB body consumes memory
  until OOM.
- **Impact:** Memory exhaustion DoS on the daemon.
- **Recommendation:** `r.Body = http.MaxBytesReader(w, r.Body, 1<<20)`
  (1 MiB is generous for AP activities).
- **Effort:** 0.1 day.

### H-6. No HSTS header
- **Location:** `internal/web/middleware.go:38-46`.
- **Attack:** `securityHeaders` sets CSP, X-CTO, X-Frame-Options,
  Referrer-Policy, but NOT `Strict-Transport-Security`. Without
  HSTS, a user typing the bare domain or following an HTTP link
  is vulnerable to TLS-stripping.
- **Impact:** TLS downgrade on first contact. Compounded by C-5.
- **Recommendation:** Add
  `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`
  when serving over TLS. Gate on `r.TLS != nil` or a config flag.
- **Effort:** 0.1 day.

### H-7. `linkify` does not strip `javascript:` URLs
- **Location:** `internal/web/server.go:478-489`.
- **Attack:** `linkify` is used in event descriptions and any
  user-supplied text rendered as HTML. The regex matches
  `https?://...` only, so `javascript:alert(1)` would NOT be
  auto-linked — but the surrounding text is `template.HTML(esc)`,
  and if the description contains
  `<a href="javascript:alert(1)">click</a>` literal markup, the
  HTML escape does NOT escape inside an existing tag (it escapes
  the `<` and `>` to `&lt;` and `&gt;`, which is safe; this
  attack does not work because the description is escaped).
  The risk is lower than first appears, but the helper is a
  footgun: any future caller that passes un-escaped text into
  linkify will open XSS.
- **Impact:** Defense-in-depth, not directly exploitable today.
- **Recommendation:** Rename the helper `safeLinkify` to flag
  the contract; add a unit test that feeds in
  `<script>alert(1)</script>` and asserts the output is
  `&lt;script&gt;...&lt;/script&gt;` (i.e. escaped). Add a
  `javascript:` rejection to the URL regex.
- **Effort:** 0.25 day.

### H-8. JSON-LD embedded as `template.JS` (raw)
- **Location:** `internal/web/handlers.go:256-259`,
  `internal/web/server.go:717-748`.
- **Attack:** `eventJSONLD` marshals a map and returns `string(b)`.
  The handler then passes it to the template as `template.JS` which
  bypasses Go's html/template escaping. The marshal step
  JSON-escapes, so `"` in the title becomes `\"`, but a description
  containing `</script>` will close the JSON-LD `<script>` tag
  in the HTML, opening the door to XSS. `e.Title` and
  `e.Description` come from the organizer's input (validated
  for length only, not content).
- **Impact:** Stored XSS via the organizer's event description
  (the attacker is the organizer in this case — the stored XSS
  then hits every visitor of the event page).
- **Recommendation:** Either escape the JSON-LD string before
  embedding (replace `</` with `<\/` per OWASP JSON-in-HTML
  guidance), or use the text/template path with
  `json.Marshal` + safe-embed helper.
- **Effort:** 0.25 day.

### H-9. Ticket purchase has no auth or rate limit
- **Location:** `internal/web/handlers.go:349-423`.
- **Attack:** `/events/{group_key}/{event_id}/purchase` accepts
  `email` and `ticket_id` from the form, no CSRF (the route is
  registered with `POST` so CSRF middleware does fire, but the
  form on the public event page must include the token), no
  rate limit, and no captcha. An attacker can buy all available
  tickets with attacker-controlled email addresses, scalping
  inventory.
- **Impact:** Inventory abuse; financial damage if the tickets
  are real-money (Stripe session is created).
- **Recommendation:** Per-IP rate limit on `/purchase` (e.g.
  1 req/s, burst 5). Add an email-confirmation step before
  charging. Consider a captcha for high-value tickets.
- **Effort:** 0.5 day.

## MEDIUM

### M-1. `ipLimiters` map grows unbounded
- **Location:** `internal/web/server.go:46-74`.
- **Attack:** Every distinct source IP gets a bucket; the map
  has no eviction. A botnet spraying `/my-rsvps/login` will
  grow the map without bound, eventually OOMing the daemon.
- **Recommendation:** LRU cap (e.g. 100k entries with TTL
  sweep).
- **Effort:** 0.5 day.

### M-2. `order_id` and `event_id` are 64-bit (truncated from 128)
- **Location:** `internal/web/handlers.go:393,1111,1136`.
- **Attack:** `generateToken()[:16]` is 64 bits. The comment
  says these IDs are "cosmetic" and the real credential is the
  `rsvp_token` (256 bits). But: the `order_id` IS the credential
  for `/checkout/{order_id}` (see H-3); the `event_id` is the
  URL parameter every public page uses and the
  `attended`-by-email lookup key in `MarkRsvpAttended` — an
  attacker who guesses an event_id can check in any RSVP at
  that event (handled at H-3 endpoint level too, but defense
  in depth: make the IDs 128 bits).
- **Recommendation:** Use `generateHighEntropyToken()[:32]`
  for `order_id` and `event_id`. Keep `recurrence_id` at 16
  bytes (it's not a security boundary).
- **Effort:** 0.1 day.

### M-3. Anti-dox rate limit is per-IP only
- **Location:** `internal/web/handlers.go:521-525`.
- **Attack:** 0.2 req/s, burst 5, per IP. A botnet of N IPs
  gives 0.2 × N req/s. To enumerate a 1M-address list takes
  ~58 days at 200 IPs; feasible for a motivated attacker.
- **Recommendation:** Add a per-email cooldown (e.g. 1
  magic-link per email per 60s, 5 per hour) on the storage
  layer. The current flow overwrites the prior session but
  still sends a fresh email every time.
- **Effort:** 0.5 day.

### M-4. Magic-link token comparison in `handleCancelRsvp` not constant-time
- **Location:** `internal/web/handlers.go:596-599` and
  `internal/web/store.go` `GetRsvpByToken`.
- **Attack:** `s.store.GetRsvpByToken(rsvpToken)` does
  `SELECT ... WHERE token = ?`. SQLite's `=` operator
  short-circuits on the first byte that differs; timing
  differences are microseconds and the per-row token
  space is 256 bits, so this is **not exploitable in
  practice** over the network. But the codebase already
  has `subtleEqual` (middleware.go:137-146) and uses it
  for CSRF; using it here is consistent.
- **Recommendation:** Use `subtle.Equal` in
  `GetRsvpByToken`. The constant-time benefit is mostly
  aesthetic for 256-bit tokens but it matches the codebase
  pattern.
- **Effort:** 0.1 day.

### M-5. Body size not capped on public POST endpoints
- **Location:** `internal/web/handlers.go:269-343
  (handleRsvpSubmit)`, `349-423 (handlePurchaseTicket)`,
  `807-836 (handleLoginPost)`, `873-968 (handleCreateGroup)`,
  `1060-1228 (handleCreateEvent)`, `1231-1279
  (handleCreateTicket)`, `1282-1311 (handleCheckIn)`.
- **Attack:** No `http.MaxBytesReader` is installed
  anywhere. A POST with a 100MB body is fully buffered by
  the form parser. CSRF middleware checks happen first
  (only the form field, not body size).
- **Impact:** Memory DoS via large POST.
- **Recommendation:** Wrap the body in
  `http.MaxBytesReader` (e.g. 64 KiB for forms, 1 MiB for
  JSON if any) at the top of each POST handler, or in
  the CSRF middleware.
- **Effort:** 0.5 day.

### M-6. `/my-rsvps/login` storage failure is silently swallowed
- **Location:** `internal/web/handlers.go:540-544`.
- **Attack:** `log.Printf` on storage failure, but the
  user sees the same "check your email" page. An attacker
  who can inject DB errors (e.g. via a crafted
  collision-prone email that overflows a TEXT column, if
  any) gets the same response. Low impact, but the
  pattern hides bugs.
- **Recommendation:** If the storage error is non-retryable
  (e.g. schema mismatch), surface a 500. For transient
  errors, the current behavior is fine.
- **Effort:** 0.1 day.

### M-7. Session token not bound to UA / IP
- **Location:** `internal/web/handlers.go:807-836` and
  `internal/web/store.go:614-640`.
- **Attack:** A stolen session cookie works from any IP /
  any browser. Combined with H-3 (no HSTS) and the
  `Secure: false` cookie (C-5), a network-sniffed session
  is fully replayable.
- **Recommendation:** Bind the session token to a UA hash
  and a /24 IP prefix at create time; reject on mismatch.
  Or move to a fingerprint cookie set in the login form.
- **Effort:** 0.5 day.

### M-8. ActivityPub actor endpoint discloses internal group data
- **Location:** `internal/activitypub/activitypub.go:172-203`.
- **Attack:** `/ap/actor/{name}` is public; the response
  includes the group's `display_name` and `summary`
  (description) verbatim. Useful for reconnaissance.
- **Impact:** Information disclosure (low sensitivity).
- **Recommendation:** No fix needed if the group page
  itself is public; the AP actor is meant to be public.
  Document the disclosure as expected.
- **Effort:** 0.

### M-9. Dockerfile uses floating base-image tags
- **Location:** `Dockerfile` (line 1: `FROM golang:1.25-alpine`;
  line 11: `FROM alpine:3.24`).
- **Attack:** A malicious upstream patch to `golang:1.25-alpine`
  or `alpine:3.24` would land in every rebuild. Dependabot
  alerts cover CVE fixes but not malicious maintainer
  takeovers.
- **Recommendation:** Pin to digest:
  `FROM golang:1.25.5-alpine@sha256:...` and
  `FROM alpine:3.24@sha256:...`. Renovate/Dependabot can
  bump the version while keeping the digest pinned.
- **Effort:** 0.1 day.

### M-10. `buildRRULE` injection surface
- **Location:** `internal/web/handlers.go:998-1057`.
- **Attack:** `recurrence_type` and `recurrence_until` are
  taken from the form and concatenated into an RRULE
  string. The `endsAt` field is parsed via `time.Parse` so
  injection is bounded, but `recurrence_type` is checked
  against a literal set of 3 values — that's fine. The
  COUNT field is parsed via `strconv.Atoi` and falls
  through silently on bad input. No direct exploit, but
  the helper has no length cap and the `endsAt` is not
  capped to a sane range (a 100-year-in-the-future date
  expands the recurrence set with 2-year lookahead —
  bounded, but worth noting).
- **Recommendation:** Cap RRULE expansion to the protocol
  limit (e.g. 1000 instances or 2 years, whichever is
  smaller — the current 2-year cap is the right default;
  document it).
- **Effort:** 0.1 day.

### M-11. `handleGroupICS` does not escape newlines in event title/desc
- **Location:** `internal/web/handlers.go:1481-1549`.
- **Attack:** The `icsEscape` helper does escape `\n` to
  `\\n`, but the DESCRIPTION field can still contain
  commas / semicolons that are escaped. The current
  helper covers all RFC 5545 special characters. **Not
  exploitable** — the helper is correct. Note for the
  next audit: a `\r` would not be escaped; the helper
  should also strip `\r`.
- **Recommendation:** Add `\r` to `icsEscape`.
- **Effort:** 0.1 day.

### M-12. `handleGroup` shows RSVP count publicly
- **Location:** `internal/web/handlers.go:189-198`.
- **Attack:** Public group page shows `RsvpCount` for each
  upcoming event. An attacker scraping this can count
  attendees per event. For privacy-sensitive groups this
  is unwanted.
- **Impact:** Reconnaissance (low — the public event page
  also shows the count via JSON-LD).
- **Recommendation:** Add a per-group "hide attendance

- **Effort:** 0.5 day.

## LOW

### L-1. PII in logs
- **Location:** `internal/web/handlers.go:421,543,558,664,
  695,963,1180,1276,1309,1680` and many more in
  `internal/email/sender.go:88` (LogOnlySender).
- **Attack:** Email addresses, RSVP tokens (via email
  content), organizer tokens, event titles, attendee
  names are written to stdout. The k8s `logOnly` mode
  prints the full email body. Anyone with read access
  to the journal (operators, log aggregator) sees PII.
- **Recommendation:** Redact emails in non-debug logs
  (use a short hash for correlation). The LogOnlySender
  should at minimum not log the body unless DEBUG is
  set. The `organizer token` log at handlers.go:964 is
  already covered in C-4.
- **Effort:** 0.5 day.

### L-2. Error messages leak internal details
- **Location:** `internal/payment/webhook.go:72`
  (signature verification error string sent to client),
  `internal/web/handlers.go:604,1119` (event creation
  error string).
- **Recommendation:** Generic error to the client; full
  error in the log.
- **Effort:** 0.25 day.

### L-3. CSRF token has 30-day MaxAge with no rotation
- **Location:** `internal/web/middleware.go:75`.
- **Recommendation:** Rotate on login / privileged
  action; or shorten MaxAge to 24h.
- **Effort:** 0.25 day.

### L-4. `subtleEqual` is hand-rolled
- **Location:** `internal/web/middleware.go:137-146`.
- **Recommendation:** Use `crypto/subtle.ConstantTimeCompare`
  (requires `[]byte`, easy wrap). The hand-rolled
  version is correct but `subtle.Equal` is the stdlib
  idiom.
- **Effort:** 0.1 day.

### L-5. `sanitizePath` is defined but never called
- **Location:** `internal/web/middleware.go:182-187`.
- **Recommendation:** Either remove or wire into the
  URL-parameter handlers (none currently use it).
- **Effort:** 0.

### L-6. CSP does not include `frame-ancestors`
- **Location:** `internal/web/middleware.go:13-17`.
- **Recommendation:** Add `frame-ancestors 'none'` to
  CSP (X-Frame-Options is set but redundant when CSP
  covers it).
- **Effort:** 0.1 day.

### L-7. `go list -m -u all` shows outdated indirect deps
- **Location:** `go.mod`. Notable: `golang.org/x/net`
  v0.56.0 is recent; `golang.org/x/sys` v0.46.0; check
  for CVEs at next Dependabot pass.
- **Recommendation:** Run `go list -m -u all` and
  bump in a separate PR. (I did not run the command;
  listed from go.mod contents.)
- **Effort:** 0.5 day.

### L-8. `net/smtp` is deprecated
- **Location:** `internal/email/sender.go:14,73`.
- **Recommendation:** Migrate to `github.com/wneessen/
  go-mail` or a maintained SMTP client. Go's stdlib
  docs explicitly warn that `net/smtp` is not actively
  maintained.
- **Effort:** 1 day.

### L-9. `internal/mcp` exposes write tools — auth model unclear
- **Location:** `internal/mcp/` (5 files, not fully
  read in this audit).
- **Attack:** MCP write tools (`internal/mcp/
  write_tools.go`) appear to bypass the organizer
  session if served on the same port. This needs a
  dedicated audit pass.
- **Recommendation:** Schedule a follow-up audit of
  the MCP surface.
- **Effort:** 0.5 day for the audit, then fixes
  per findings.
